ORB-SLAM3 final backend stability pass
Date: 2026-06-09

Purpose:
- Continue after the interrupted run.
- Make the custom true-timestamp ORB-SLAM3 runner stable enough for the next data-collection round.
- Avoid the two known backend crash points.

Code changes kept:
- raspi_mono_inertial_slam.cc
  - Direct IMU CSV input.
  - Real image timestamp input.
  - SIGSEGV/SIGABRT backtrace handler.
- Optimizer.cc / Optimizer.h
  - PoseOptimization now rejects bad/insufficient inputs before building a g2o graph.
  - PoseOptimization checks finite frame pose, finite map points, valid keypoint octaves, valid edge counts, and optimizer return values.
  - The main mono-inertial InertialOptimization now returns bool instead of void.
  - InertialOptimization checks keyframe count, valid IMU preintegration dt/covariance, valid edge count, finite scale/bias/gravity output.
  - The inertial optimizer graph is allocated on the heap to avoid this fork's g2o destructor double-free path.
- LocalMapping.cc
  - IMU initialization is disabled by default for the Raspberry Pi export path.
  - To explicitly experiment with the unstable inertial backend, set:
    RASPI_ORBSLAM3_ENABLE_IMU_INIT=1
  - If enabled, IMU initialization is still gated by time/keyframe/motion/preintegration checks.
  - CreateNewMapPoints is enabled, so the map exports real triangulated ORB points.
- Tracking.cc
  - Before IMU initialization, the unsafe visual PoseOptimization path is skipped for IMU_MONOCULAR.

Why IMU initialization remains disabled by default:
- With current data, the first crash is in visual PoseOptimization:
  EdgeSE3ProjectXYZOnlyPose::linearizeOplus / g2o optimization.
- If that path is skipped, the next crash is in inertial initialization:
  LocalMapping::InitializeIMU -> Optimizer::InertialOptimization -> g2o double free.
- The inertial optimizer is still unsafe in this local ORB-SLAM3 fork.
- A robust production behavior is to avoid killing the process and allow full export.
